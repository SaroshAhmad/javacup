# ADR 0007 — Logged-in home ("dashboard")

- **Status:** Accepted — deferred to Phase 2 (per PRD MVP v2). Build-ready spec captured here.
- **Date:** 2026-06
- **Related:** PRD MVP v2 (§3 Deliberately Excluded → Phase 2: "Roadmap progress tracker"),
  ADR 0005 (roadmap vision), screen-inventory.md, sitemap.md

---

## Context

The question arose: if the MVP has no dashboard, what does login unlock, and where does a
logged-in user go after signing in? Investigation clarified two things:

1. **Login unlocks actions, not a place.** JavaCup is read-only for Guests and
   participatory for Members. Login enables voting, posting recommendations and
   discussions, suggesting resources, flagging, and carrying an identity tag on every
   contribution. None of this requires a dashboard — it happens on the public pages. So
   auth is fully justified by participation alone.

2. **There is a real gap in the post-login experience.** Today login redirects to `/`
   (the marketing homepage), which still shows "Join for free" CTAs that no longer apply.
   Landing a member back on the join pitch is flat and a small missed retention moment.

A richer idea was explored: a **logged-in home base** — "welcome back, here's your next
step, your activity, what's trending" — distinct from a settings/management panel. It
tested well as a concept and fits JavaCup's identity (it answers *"what do I do next?"*
rather than showing vanity metrics).

**However:** the PRD MVP v2 already, deliberately, defers the **Roadmap progress tracker
to Phase 2** (trigger: MVP success metrics met, 100 active users, ~Q4 2026). The logged-in
home depends on exactly that progress data ("continue where you left off", "Core Java
60%"). Pulling the home into the MVP would force pulling the deferred progress tracker
forward too — unwinding a considered sequencing decision and growing an already-large
(~32 screen) MVP. The PRD's own rule applies: *"Never build outside it without updating it
first."*

## Decision

**Keep the logged-in home (dashboard) in Phase 2, consistent with the PRD.** Do not pull
it or its dependency (progress tracking) into the MVP.

**Interim, in-scope solution for the post-login gap:** redirect login to the **Roadmap**
(`/roadmap`) instead of the marketing homepage. The roadmap is the core "what's next"
surface, is useful to a returning learner, and needs no new features. This gives login a
clear payoff today at zero scope cost. (The logged-in navbar already swaps to the profile
menu, so members already get an authenticated-state signal everywhere.)

When Phase 2 opens, the logged-in home becomes the natural post-login destination,
replacing the roadmap redirect.

## Phase 2 spec (build-ready)

A **logged-in home base** — a personalized landing page, NOT a settings/management panel.
Its job: answer "what do I do next?" and reflect the user's place in the community.

### Dependencies (must exist first, all Phase 2 or earlier)
- **Progress tracking** (Phase 2) — users mark topics/stages complete; per-user state;
  computed stage percentages. This is the gating dependency.
- **Activity data** — counts of posts, replies, recommendations, votes cast, trust earned.
  Most of this is generated by MVP features (F-02/F-03/F-04); only the aggregation view is new.
- **"Recommended next step" logic** — derive the next topic from progress + roadmap order.

### Layout (top to bottom), in brand (warm-dark, amber accent, restraint)
1. **Welcome header** — "Welcome back, {name}" + the user's identity-tag pill + a one-line
   orientation ("You're on Core Java · stage 2 of 5").
2. **Primary card — "Continue where you left off"** — the single most important element.
   Mono eyebrow `// continue where you left off`, the next topic name, its priority + how
   many resources/recommendations it has, and ONE amber primary "Continue →" button. Faint
   dot texture. This is what makes it a *guidance* home, not a generic dashboard.
3. **Roadmap progress** — the 5 stages with done/in-progress/locked states (green done,
   amber current with %, muted upcoming).
4. **Your activity** — posts, votes cast, recommendations, and **trust earned** (green) as
   simple stat blocks. Ties to the voting/reputation core.
5. **Trending in the community** — top voted recommendations/discussions with identity pills
   and vote scores. Keeps the community pulse visible; pulls the user back in.
6. **Contribute nudge** — a quiet strip linking to the contributor/cohort vision.

### Explicitly NOT included
- No vanity charts or metric walls. Restraint is the brand.
- Not a settings panel — profile editing stays on the Profile page (F-05).
- Not "management" — there is little to manage; the value is direction + community pulse.

### Route
`/home` or `/feed` (member-only, redirect to `/login` if unauthenticated). Becomes the
post-login destination, replacing the interim roadmap redirect.

## Consequences

**Positive**
- MVP scope and sequencing stay intact; the disciplined "don't grow the MVP" line holds.
- The post-login flatness is fixed now, cheaply (redirect to roadmap).
- The dashboard vision is captured with a concrete, considered spec — it becomes a strong
  *first* Phase 2 build, not a vague "someday", and not lost.
- Building it in Phase 2 means the progress/activity data exists to fill it — no empty shell.

**Negative / watch**
- "Logged-in home" must not creep back into the MVP by accident; this ADR is the record
  that it was considered and deferred, with reasons.
- When built, it depends on progress tracking being done first — sequence accordingly
  within Phase 2 (tracker → home).

## Notes
This ADR reverses an in-the-moment instinct to add the dashboard to the MVP, after finding
the PRD already deferred its dependency to Phase 2. Recording it so the reasoning is
durable and the feature is build-ready when the time comes.
